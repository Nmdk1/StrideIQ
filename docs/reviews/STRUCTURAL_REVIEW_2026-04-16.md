# StrideIQ Structural Review — April 16, 2026

This is a structural assessment of the StrideIQ backend as it exists today, read directly from source. It is not a product review and contains no feature recommendations. Every claim cites file paths and, where relevant, line counts verified at the time of this reading.

The recent refactor moved roughly 33 service modules from the top level of `apps/api/services/` into three domain packages (`sync/`, `intelligence/`, `coaching/` + `coach_tools/`) and split `models.py` into a `models/` package. Every moved module left behind a four-to-seven-line import shim so old call sites (`from services.strava_service import X`, `from services.correlation_engine import Y`, `from services.ai_coach import Z`) still resolve. That work is cleanly executed and the shim pattern is consistent. What follows examines what the refactor did, what it left behind, and what should happen next.

## Section 1 — Current architectural shape

The system is a FastAPI monolith fronted by Caddy, backed by PostgreSQL 16 with TimescaleDB, Redis, and MinIO. A single Uvicorn process with four workers handles HTTP, with Celery workers and a Celery Beat container handling background work. Everything lives on one Hostinger droplet. The web frontend (`apps/web/`) is a separate Next.js 14 app behind the same Caddy and is not the subject of this review.

The backend (`apps/api/`) orients around six layers.

The **entry layer** is `main.py`, which registers 60+ routers at module import time. Most registrations are direct imports at the top of the file; a handful (Garmin, GDPR, public tools, Runtoon, RPI, knowledge) are wrapped in `try/except ImportError` and log a warning on failure. That pattern is benign in theory but means a broken router can silently vanish from the API in production with only a log line.

The **core layer** (`apps/api/core/`) contains 19 modules: `database.py`, `security.py`, `auth.py`, `config.py`, `cache.py`, `llm_client.py`, `rate_limit.py`, `security_headers.py`, `feature_flags.py`, `logging.py`, `exceptions.py`, and a handful of smaller utilities (`date_utils`, `tier_utils`, `password_policy`, `bmi_config`, `events`, `pace_access`, `account_security`). This is the one layer that looks right — small, cohesive, no domain logic mixed in. Note that `apps/api/database.py` at the repo root is a seven-line deprecation shim pointing at `core.database`; it still exists because some older imports have not been migrated.

The **model layer** (`apps/api/models/`) is the post-refactor state of what used to be a 3,426-line `models.py`. It is split into eight domain files plus an `__init__.py` that re-exports all 87 ORM classes for backward compatibility:

- `athlete.py` — Athlete, goals, overrides, calibration, photos, learning (13 classes)
- `activity.py` — Activity, splits, streams, PBs, best efforts, feedback, stream cache (10 classes)
- `plan.py` — TrainingPlan, PlannedWorkout, templates, calendar, adaptation proposals (14 classes)
- `checkin.py` — DailyCheckin, BodyComposition, WorkPattern, IntakeQuestionnaire, DailyReadiness (5 classes)
- `nutrition.py` — NutritionGoal, NutritionEntry, USDAFood, FuelingProduct, MealTemplate (5 classes)
- `coaching.py` — CoachChat, recommendations, intent snapshots, usage, action proposals (7 classes)
- `correlation.py` — findings (both legacy and canonical), lifecycle logs, auto-discovery tables, suppression (16 classes)
- `system.py` — billing, Stripe, invites, audit, ingestion state, telemetry, runtoon, experience audit (17 classes)

The domain split is good. Two small smells: `correlation.py` carries both the canonical `CorrelationFinding` and an older `AthleteFinding` whose role relative to the canonical table is not documented in code, and `system.py` is a grab bag that has started to act like a miniature `models.py` for anything that doesn't fit elsewhere (Stripe, telemetry, Runtoon images, experience audit).

The **service layer** (`apps/api/services/`) is the heart of the application and the least settled part of the structure. It contains eleven subpackages and roughly 120 top-level `.py` files (excluding the 33 shims from the refactor). The subpackages are:

- `coaching/` — ten modules, Mixin-composed `AICoach` class (see Section 5)
- `coach_tools/` — nine modules, the functions the coach calls as tools
- `coach_modules/` — three modules (`routing.py`, `context.py`, `conversation.py`), an earlier extraction layer that `coaching/core.py` still imports from directly
- `intelligence/` — 16 modules, the correlation engine and its narrative outputs (see Section 4)
- `plan_framework/` — 19 modules, the V1 ("N1 Engine V3") plan generator
- `plan_engine_v2/` — 13 modules, the V2 plan generator running behind `engine=v2`
- `sync/` — 16 modules, Strava/Garmin/FIT ingestion, deduplication, backfill, OAuth
- `auto_discovery/` — six modules, the nightly correlation discovery pipeline
- `data_streams/` — three modules (`base.py`, `models.py`, `registry.py`), thin abstractions
- `research_data/` — three modules (age grading, figshare processing, population comparison)
- `provider_import/` — a single file (`garmin_di_connect.py`); the package is effectively underpopulated

The top-level `services/` directory still contains substantial files that never moved into a subpackage. Ten of them are over 1,000 lines and a handful are well beyond that. The largest are `race_input_analysis.py` (2,916 lines), `model_driven_plan_generator.py` (2,082), `run_stream_analysis.py` (1,663), `contextual_comparison.py` (1,552), `shape_extractor.py` (1,512), `workout_prescription.py` (1,500), `run_analysis_engine.py` (1,238), `fitness_bank.py` (1,172), `workout_classifier.py` (1,161), `experience_guardrail.py` (1,117), `operating_manual.py` (1,109), `training_load.py` (1,071), and `constraint_aware_planner.py` (1,052). These are the files that the current refactor did not touch.

The **router layer** (`apps/api/routers/`) is 60 modules. Most are small and route-focused. Five are not: `home.py` (3,613 lines), `plan_generation.py` (2,425), `admin.py` (2,317), `progress.py` (2,090), and `calendar.py` (1,384). Section 2 has more to say about these.

The **task layer** (`apps/api/tasks/`) is 18 modules covering Celery work: Strava/Garmin sync, home briefing generation (the Lane 2A worker), correlation sweeps, auto-discovery, plan lifecycle, intelligence rules, fact extraction, Runtoon generation, and best-effort post-processing. Scheduling is centralized in `celerybeat_schedule.py` and supplemented by `beat_startup_dispatch.py`, which dispatches daily tasks on container start if they have not run in the last 20 hours. That second file exists because the beat container is recreated on every deploy, and deploys happen before the 4–5 AM daily task window. Without it, production would miss daily tasks on any day the system was deployed; with it, the system is deployment-proof. It is the most important structural fix in the task layer and is not something you would guess from a cursory read of the codebase.

The data flow, end to end, is: provider webhook (Garmin or Strava) arrives at a webhook router, queues a Celery task in `sync/`, which persists an `Activity` with `ActivitySplit` and `ActivityStream`, stamps it with wellness data from `services/wellness_stamp.py`, and triggers downstream correlation input aggregation. The nightly `auto_discovery` orchestrator scans candidates, promotes confirmed patterns into `CorrelationFinding`, and the fingerprint bridge translates those findings into `FingerprintParams` that V1 and V2 plan engines can consume. The home briefing worker (`tasks/home_briefing_tasks.py`) pulls all of that context together and calls an LLM to produce the daily briefing, which the request path reads from Redis. The AI coach runs on the request path, pulls context via the `coach_tools/` functions, and streams back to the athlete. That is the system.

## Section 2 — Coupling analysis

The refactor's domain boundaries are clean where they are drawn but the hottest coupling lives in places the refactor did not touch.

The cleanest boundary is `services/sync/`. Each file is a provider or utility adapter and the package only imports from `core/`, `models/`, and a few leaf services (`services/efficiency_calculation.py`, `services/mileage_aggregation.py`). Nothing in `sync/` reaches into `intelligence/` or `coaching/`. The boundary holds.

The next cleanest is `services/plan_engine_v2/`. Its `engine.py` takes `FitnessBank`, `FingerprintParams`, and `LoadContext` as inputs and is explicit about being decoupled from the database — the router adapter loads those objects from DB and passes them in. The package imports from `services/fitness_bank.py`, `services/plan_framework/fingerprint_bridge.py`, and `services/plan_framework/load_context.py` but nothing else. For a new subsystem that will eventually replace an older one, this is exactly the right shape.

The `coaching/` and `coach_tools/` boundary is also sound in direction: `coaching/` imports `coach_tools`, not the other way around. But the coaching package itself reaches into three other places. `coaching/core.py` imports `MessageRouter`, `MessageType`, `ContextBuilder`, and `ConversationQualityManager` from `services.coach_modules` — an earlier extraction that the refactor did not fold in. `coach_modules/` is in effect a third package alongside `coaching/` and `coach_tools/`, and its role is not visible from naming alone. The coach surface is now a three-package domain pretending to be two.

The real coupling problem is not between the refactored packages; it is between the routers and everything else. `routers/home.py` has three actual HTTP endpoints (`GET /v1/home`, `GET /v1/home/signals`, `POST /v1/home/admin/briefing-refresh`) and 35+ private helper functions above them, including `generate_coach_home_briefing` (the 500+ line LLM briefing builder), `compute_coach_noticed`, `_build_rich_intelligence_context`, `_summarize_workout_structure` (the 5-gate workout structure detector), `_fetch_llm_briefing_sync`, `_call_opus_briefing_sync`, `_call_gemini_briefing_sync`, `_normalize_cached_briefing_payload`, `_valid_home_briefing_contract`, `_validate_briefing_diversity`, `validate_sleep_claims`, `_strip_ungrounded_sleep_sentences`, `validate_voice_output`, `validate_relative_time_claims`, `_build_sleep_baseline_guidance`, `_get_race_forecast`, `_get_personal_heat_multiplier`, `_build_race_weather_context`, `compute_race_countdown`, and `compute_last_run`. Every one of those is a service-layer concern living in a router. The wiki's own note that the briefing is "the most fragile surface" and has caused "repeated regressions" is consistent with this: the briefing is simultaneously the system's most complex prompt and its most coupled piece of code, and it lives in the HTTP layer.

`routers/plan_generation.py` (2,425 lines, 22 routes) has similar but less severe coupling. The route handlers are shallow but the file also holds `_save_plan`, `_save_model_driven_plan`, `_save_constraint_aware_plan`, `_generate_model_insights`, `_resolve_variant_for_constraint_workout`, and the pace-order-violation validators for each of the three plan generation pathways. Save logic and validator logic that spans three engines is natural service material, not router material. That it sits here is how the file got to 2,425 lines.

`routers/progress.py` (2,090 lines, 8 routes) is the Personal Operating Manual's assembly layer and it reads as one. `_assemble_verdict_data`, `_assemble_chapters_data`, `_assemble_patterns_data`, `_assemble_looking_ahead`, `_assemble_data_coverage`, `_assemble_knowledge`, `_generate_narrative_llm`, `_apply_llm_narratives`, `_generate_progress_headline`, `_generate_knowledge_llm` — the entire Operating Manual page is built inside the router, calling the LLM twice along the way. The actual HTTP endpoints are thin wrappers over these assemblers.

`routers/admin.py` (2,317 lines, 45 routes) is genuinely big but is also genuinely an admin router: its handlers are mostly small, each wraps an operational action (grant trial, revoke VIP, enqueue Garmin backfill, run a correlation test, issue a race promo code), and helpers are limited to the ones that make sense for admin flows (`_ensure_flag_exists`, `_utcnow`). This file is long because the admin surface is wide, not because it has absorbed service logic.

There are no observed circular imports among the refactored packages. `services/plan_framework/fingerprint_bridge.py` is the explicit bridge from `intelligence/` to `plan_framework/`; it is imported by both V1 and V2 plan engines and by nothing in `intelligence/`. That direction holds.

Three smaller couplings are worth naming. First, `services/intelligence/correlation_engine.py` defines two large module-level dictionaries — `CONFOUNDER_MAP` and `DIRECTION_EXPECTATIONS` — that encode partial-out variable selection and expected directional sign for specific `(input, output)` pairs. These are hard-coded at the top of a 2,822-line module. They grow every time a new input is added. Second, `services/efficiency_calculation.py` is imported from `intelligence/correlation_engine.py`, from `plan_framework/`, from `coaching/`, and from several analysis modules; it is a high-fan-in utility that has quietly become the canonical efficiency computation. That is probably correct but it is not documented as such. Third, `services/n1_insight_generator.py` holds the `OUTPUT_METRIC_REGISTRY` that governs which metrics are allowed to make directional claims to the athlete (the Athlete Trust Safety Contract). This registry is the authority for several services (`causal_attribution`, `coach_tools`, `load_response_explain`, `efficiency_analytics`), and the workspace cursor rule explicitly identifies those services as still carrying local polarity assumptions that should defer to the registry. That's documented migration debt, but it means the contract's enforcement is currently distributed across six files.

## Section 3 — God file watch

The refactor retired the old giants. `models.py` no longer exists at the repo root. `ai_coach.py` at the service top level is a seven-line shim to `services.coaching`. `coach_tools.py` as a single file is gone; the package replaces it. The wiki's claim that `ai_coach.py` is still "a 5.7K line god file, split deferred" and `coach_tools.py` is "4.9K lines, split deferred" is stale. The split happened. The wiki's snapshot did not follow.

Thirteen production source files exceed 1,500 lines as of this reading. Here is the full list, with a read on each.

`routers/home.py` at 3,613 lines is trending toward god-file status and is the most urgent of the group. It is not a router; it is a home page orchestration module pretending to be one. Natural seams are obvious from the function grouping: the LLM briefing builder and its two model-specific callers, the workout structure detector, the briefing validators, the race weather/countdown helpers, the compute-coach-noticed logic, and the three HTTP handlers. Splitting would pull roughly 2,000–2,500 lines into a dedicated briefing service.

`services/race_input_analysis.py` at 2,916 lines is the next largest. It is race history analysis — predicting race outcomes from recent training inputs. I did not read this file end-to-end but its size puts it on the watchlist; the natural seams, if splitting becomes necessary, would likely be by input domain (pace history, training load, wellness) rather than by function type.

`services/intelligence/correlation_engine.py` at 2,822 lines is the brain of the system and is discussed at length in Section 4. It is trending toward god-file status but its internal organization (40+ narrowly-scoped `aggregate_*` functions, each one an input builder, plus a smaller set of correlation/gating/discovery functions) is coherent. If it needs splitting, the natural seam is between input aggregation (~1,700 lines) and correlation/discovery (~1,100 lines).

`routers/plan_generation.py` at 2,425 lines is trending toward god-file status but the specific problem is not size — it is that the file holds the save logic for three different plan engines. See Section 6.

`routers/admin.py` at 2,317 lines is fine. Admin surfaces accrete routes; 45 small handlers in one file is manageable and the alternative (five admin routers) creates a different kind of confusion. Leave it alone.

`routers/progress.py` at 2,090 lines is trending toward god-file status. The Operating Manual assembly logic belongs in a service; the LLM narrative generation is the highest-value piece to extract because it is called from two places (progress and briefing) with similar-but-not-identical contracts.

`services/model_driven_plan_generator.py` at 2,082 lines is not a god file; it is a legacy file that should be dead code. See Section 6.

`tasks/strava_tasks.py` at 1,787 lines is borderline. Strava sync is inherently complicated — rate limiting, backfill, deduplication, webhook correlation, PB detection — and 1,787 lines for the full surface is not out of line for that complexity. Watch, don't split.

`services/run_stream_analysis.py` at 1,663 lines, `services/contextual_comparison.py` at 1,552, `services/shape_extractor.py` at 1,512, and `services/workout_prescription.py` at 1,500 are all over the 1,500 threshold but none is trending toward god-file status in the way `home.py` is. Each is a single-purpose module that happens to be large because the purpose is large. They are fine.

`services/plan_engine_v2/workout_library.py` at 1,627 lines deserves a separate note. It is the workout catalog for V2 and is almost entirely data — 22+ workout types with concrete segment definitions. Its size is proportional to its role and splitting by workout family (easy/moderate, threshold, interval, long run, race-pace) would make the catalog easier to browse, but only if the catalog keeps growing. For now it is fine.

`services/coaching/_prescriptions.py` at 1,024 lines and `coaching/_llm.py` at 1,019 lines are at the threshold. These are Mixin modules on the `AICoach` class, so they can only grow if the coach's responsibilities grow. Both are single-purpose within the mixin model. Fine.

The refactor retired the three worst offenders. What remains is a tail that is mostly under control, with `routers/home.py` as the one clear exception that will benefit from a split and a handful of others that should be monitored but not touched.

## Section 4 — The correlation engine

The correlation engine is implemented across six files in `services/intelligence/` plus two thin bridges. It is a scientific instrument for one athlete, designed around the principle that no threshold is universal and that the system should compile evidence before making directional claims.

The core file is `correlation_engine.py` at `services/intelligence/`. Its public entry points are `aggregate_daily_inputs`, `aggregate_activity_level_inputs`, `aggregate_cross_training_inputs`, `_aggregate_fueling_inputs`, `aggregate_feedback_inputs`, `aggregate_training_pattern_inputs`, `aggregate_training_load_inputs`, `aggregate_pace_at_effort`, and roughly a dozen more `aggregate_*` functions — one per input domain. Each one takes an athlete and a date range, queries the relevant models, and returns a dict keyed by input name with one scalar value per observation. This is the input-builder phase. It is roughly 1,700 of the file's 2,822 lines and the pattern is consistent across every input.

The second phase is correlation. `calculate_pearson_correlation` is the bivariate primitive. `calculate_weighted_pearson_correlation` applies the temporal weighting schema documented in the wiki (L30: 4×, L31–90: 2×, L91–180: 1×, >180d: 0.75×) through a `_recency_weight` helper that reads the observation date and reference date. `find_time_shifted_correlations` iterates lags from 0 to 7 days and picks the peak. `compute_partial_correlation` applies confounder control: given an `(input, output)` pair, it consults the module-level `CONFOUNDER_MAP` dictionary for the variable to partial out, and returns the partial correlation coefficient. The map has roughly 25 entries covering the common cases (readiness → efficiency confounded by daily session stress; TSB → efficiency confounded by daily session stress to avoid circularity, since TSB is derived from CTL and ATL; sleep → pace confounded by CTL to avoid the taper artifact where athletes sleep more and run faster simultaneously; cadence → efficiency confounded by elevation gain to avoid the hilly-terrain artifact). The confounders that are not in the map get bivariate correlation unmodified. This design is deliberate: adding a new confounder requires a code change plus a test, not a configuration edit.

The gates are `|r| >= 0.3`, `p < 0.05`, `n >= 10`, and for sustained findings `times_confirmed >= 3`. The p-value comes from `scipy.stats.t.sf` — the wiki notes that this replaced several custom t-distribution approximations that were silently producing wrong p-values in `correlation_engine.py`, `efficiency_trending.py`, `causal_attribution.py`, `pre_race_fingerprinting.py`, and `trend_attribution.py`. That fix is already landed; mentioning it here because the historical presence of five hand-rolled statistical approximations is a data point about how this layer grew.

The third phase is layer analysis, in `correlation_layers.py`. Four layers are implemented and named in the code:

L1 (threshold detection) finds non-linear boundaries where the relationship between input and output changes character (efficiency drops below 6.5 hours of sleep, for example). It does this by searching for an inflection point that maximizes the difference in mean output above and below the candidate threshold, subject to minimum sample sizes on each side.

L2 (asymmetric response) tests whether above-threshold and below-threshold effects have different magnitudes. A symmetric input has equal upside and downside; an asymmetric one does not.

L3 (cascade/mediation) uses `compute_partial_correlation` to check whether the input→output relationship survives when a mediator is controlled for. If it does not survive, the mediator is the causal path.

L4 (lagged decay) fits a decay curve to time-shifted correlations, producing a best-fit lag and a decay rate. Used for things like strength-training-to-efficiency where the peak effect is several days out.

Findings that pass the gates are persisted. `correlation_persistence.py` writes them to `CorrelationFinding` with lifecycle state `emerging`. `insight_aggregator.py` is the reader: it loads the active findings, groups them by output metric, and produces the aggregated insight set that the Operating Manual, home page, and coach all consume. `n1_insight_generator.py` is a different and more authoritative reader: it owns the `OUTPUT_METRIC_REGISTRY` whitelist that governs which metrics can make directional claims to the athlete (the Athlete Trust Safety Contract, documented in its module docstring and enforced by 65 contract tests). If a finding's output metric is not in the whitelist or has ambiguous polarity, the generator emits neutral observation text; if the metadata is missing, conflicting, or invalid, it suppresses the directional output entirely. Efficiency (pace over heart rate) is the canonical ambiguous metric because the ratio moves in opposite directions for two equally valid improvement modes, and the generator is the single place that is guaranteed to get this right.

The lifecycle is managed by the Limiter Engine, a separate subsystem layered on top of findings. `services/plan_framework/limiter_classifier.py` (873 lines) reads active findings and classifies the athlete's dominant limiter into one of seven categories (L-VOL, L-CEIL, L-THRESH, L-REC, L-CON, L-SPEC, L-NONE). The classifier is rule-based for L-SPEC (race-specific) and evidence-based for the others. Transitions (`emerging` → `active` → `resolving` → `closed`, with `structural` and `structural_monitored` branches for long-standing physiological traits) are computed by re-weighting findings against recent data and checking whether the L30-weighted correlation still passes the gates.

The `AutoDiscovery` subsystem (`services/auto_discovery/`) runs nightly at 4 AM UTC and is the forward-looking research arm. `orchestrator.py` drives the pipeline; `rescan_loop.py` re-evaluates existing candidates; `interaction_loop.py` tests pairs of inputs for interaction effects; `tuning_loop.py` tunes thresholds and parameters; `fqs_adapters.py` computes Finding Quality Score. Each loop can be turned off independently via `feature_flags.py`. Models are `AutoDiscoveryRun`, `AutoDiscoveryExperiment`, `AutoDiscoveryCandidate`, `AutoDiscoveryChangeLog`, `AutoDiscoveryScanCoverage`. The wiki notes this has produced roughly 100 promoted findings, 62 stability annotations, and 20 interaction candidates as of April 6.

The fingerprint bridge — `services/plan_framework/fingerprint_bridge.py` — is where findings become plan parameters. It reads confirmed findings for an athlete, translates specific correlation outputs into plan-relevant parameters (recovery half-life, quality spacing hours, cutback frequency, dominant limiter, primary quality emphasis), and emits a `FingerprintParams` dataclass. Both V1 (`plan_framework/n1_engine.py`) and V2 (`plan_engine_v2/engine.py`) consume this. The wiki correctly notes that only `cutback_frequency` and `quality_spacing_min_hours` are currently consumed by V1 session scheduling; `limiter` and `primary_quality_emphasis` are computed but unused. That is a wiring gap, not a design gap — the contract is there, the consumer is not complete.

Three narrative-adjacent modules in `intelligence/` turn findings into words: `narrative_translator.py` (athlete-facing language translation), `narration_scorer.py` (quality gate on adaptation narratives, 46 contract tests), `adaptation_narrator.py` (produces the 2–3 sentence narration that accompanies every adaptation decision), and three others that overlap — `moment_narrator.py`, `narrative_memory.py`, `workout_narrative_generator.py`, and `neutral_terminology.py`. That proliferation is discussed in Section 5 and Section 7.

Where an engineer new to this subsystem would get stuck: the `CONFOUNDER_MAP` and `DIRECTION_EXPECTATIONS` dictionaries at the top of `correlation_engine.py` govern real athlete-facing behavior. Changes there are not cosmetic. They live in code for a reason (every change requires a test), but they are not called out structurally, and the two maps share an implicit dependency with `OUTPUT_METRIC_REGISTRY` in `n1_insight_generator.py` that only the contract tests enforce.

## Section 5 — The coach layer

The AI coach is a single `AICoach` class with seven mixins. It lives in `services/coaching/core.py` and is composed by multiple inheritance: `class AICoach(BudgetMixin, ToolsMixin, LLMMixin, ContextMixin, ThreadMixin, GuardrailsMixin, PrescriptionMixin)`. Each mixin is in a private module under the same package: `_budget.py` (enforces the per-athlete request and token caps against `CoachUsage` records), `_tools.py` (wires the coach's ~26 function-calling tools to the `coach_tools/` package), `_llm.py` (the actual LLM call — Kimi K2.5 primary, Claude Sonnet 4.6 silent fallback), `_context.py` (constructs athlete context for non-briefing surfaces), `_thread.py` (conversation thread management backed by `CoachChat` JSONB messages), `_guardrails.py` (KB violation scanner, anti-hallucination checks, date rendering discipline), `_prescriptions.py` (the workout-prescription response path). `_constants.py` holds the budget caps, the high-stakes query patterns, and a few utility functions shared across mixins.

This is a healthy pattern. Mixins decompose a large behavioral class into narrow modules while keeping a single entry point. The decomposition is by concern (budget, tools, LLM, context, thread, guardrails, prescriptions) rather than by code size, which is the correct axis.

The context builders are where things get confusing. There are three in play. `build_athlete_brief()` in `coach_tools/brief.py` is the comprehensive one — 14-day recent runs, identity, wellness, PBs, race predictions, fingerprint, weekly volume, and now a Nutrition Snapshot. Its header instructs the LLM to use pre-computed relative date labels verbatim. `_build_athlete_context_for_chat()` in `coaching/_context.py` is the chat-specific one — 7-day summary, 30-day stats, check-ins, Garmin watch data, planned workouts, with the same date-label discipline. `_build_athlete_state_for_opus()` (also in `_context.py`) is a premium path for deeper context. The wiki already identifies this as a known issue and it is correct: the first two overlap significantly and the authoritative source is not obvious without reading both.

The coach imports one more package, which is the piece the refactor didn't clean up: `services/coach_modules/`. `coaching/core.py` has `from services.coach_modules import MessageRouter, MessageType, ContextBuilder, ConversationQualityManager`. That is a separate three-file package (`routing.py`, `context.py`, `conversation.py`) that predates the `coaching/` split and is still doing real work — message classification (judgment/prescription/comparison/general), return-context detection, confidence-gated responses, answer-pattern tracking. There is no technical problem with using it; the problem is that a reader trying to understand the coach surface now has to navigate `coaching/` and `coach_modules/` and `coach_tools/` without any naming clue about the layering between them.

The prompt construction is in `coaching/core.py` `SYSTEM_INSTRUCTIONS` and in `_llm.py`'s prompt assembly. The system prompt is ~780 lines long in the class body and contains the athlete-calibration instructions, the anti-hallucination rules, the terminology constraints, the tool-use discipline, and the N=1 principles. Context is assembled in `_llm.py` by concatenating the selected builder's output (one of the three above) with the coach's function-calling tools and the user message. The LLM call goes through `services/core/llm_client.py`.

Now the finding-to-output path, which was called out as the specific thing to trace. A finding starts as a `CorrelationFinding` row in `active` state. The coach can request findings in two ways. First, through the `get_correlations` or `get_active_insights` tools in `coach_tools/insights.py`, which read the database directly. Second, through the briefing pathway, where `build_fingerprint_prompt_section()` in `services/fingerprint_context.py` assembles a curated fingerprint block that the briefing prompt inlines. The two paths produce differently-formatted text from the same underlying data.

Along the coach path, the potential for distortion is bounded by five guardrails, any one of which can fail independently. The first is the `OUTPUT_METRIC_REGISTRY` whitelist — if the finding's output metric is not registered, the insight is neutralized before it reaches the coach's context. The second is `_check_kb_violations` in `_guardrails.py`, which scans output text against the 76-rule KB registry after generation. The third is the data-verification discipline in the system prompt — the LLM is instructed to call `get_training_paces`, `get_recent_runs`, and the other tools rather than invent numbers. The fourth is the date-rendering guardrail — every date in context has a pre-computed relative label, so the LLM is not computing "yesterday" or "last week" itself. The fifth is `_check_response_quality`, which runs post-generation checks for known failure modes (fact capsule leakage, contract echoes, raw metric dumping).

The weakest point in this chain is the gap between "finding is in context" and "finding appears in output." The registry protects directional claims at context-build time, but a finding can still be paraphrased incorrectly by the LLM — a confirmed correlation with r=0.45 and n=18 can be rendered as "you always run faster when" rather than "across 18 sessions, runs with X tend to be faster by Y." Neither the KB scanner nor the response quality checker catches that class of distortion today. That is the specific place a claim can still degrade: in the language surface, not in the structural surface. The adaptation path has a narration scorer (46 contract tests in `narration_scorer.py`) that catches this class of issue for the narrower adaptation case; the general coach surface does not have an equivalent.

The briefing is a different code path from the coach. `generate_coach_home_briefing` in `routers/home.py` builds its own prompt from eight to ten intelligence sources (athlete brief, today's completed activity, workout structure summary, planned workout, check-in data, wellness trends, recent runs, fingerprint section, race countdown, cross-training context), calls Claude Opus 4.6 directly (with Gemini 2.5 Flash as the fallback), and writes the result to Redis. The coach path calls Kimi K2.5. The two systems share `build_athlete_brief`, `get_wellness_trends`, the fingerprint prompt section, and the `_relative_date` date discipline, but they do not share prompt assembly, they do not share model routing, and they do not share guardrails. The briefing has its own post-generation validators in `home.py` (`validate_sleep_claims`, `validate_relative_time_claims`, `_validate_briefing_diversity`, `validate_voice_output`). The coach's equivalent is in `_guardrails.py`. This duplication is not harmful — the two surfaces have different failure modes — but the two sets of validators have already diverged and there is no shared validator library that both consume.

## Section 6 — The next three structural moves

These are the three highest-leverage structural moves to make next, in priority order. None is a feature. None is a rewrite.

**1. Extract briefing and home orchestration out of `routers/home.py` into `services/briefing/`.** The problem this solves is concrete: the most fragile surface in the product lives in a 3,613-line router where HTTP handling, LLM orchestration, prompt assembly, workout structure detection, sleep validators, diversity validators, relative-time validators, cache management, and race weather helpers are all stitched together. The wiki is explicit that briefing regressions have happened "repeatedly" and the advisor has had to intervene after builder changes. That pattern will keep repeating as long as the briefing lives in the router because every change touches a file that also handles HTTP routing, and the shared concerns bleed across PRs.

The scope is roughly 2,000–2,500 lines moved into a new `services/briefing/` package with four or five modules: `prompt_assembly.py` (everything the current `generate_coach_home_briefing` does), `structure_detector.py` (`_summarize_workout_structure` and the 5-gate architecture — which probably belongs in `services/intelligence/` rather than `briefing/`, but I'd leave that decision to the work), `validators.py` (the sleep, relative-time, diversity, and voice validators), `llm_callers.py` (the `_call_opus_briefing_sync` and `_call_gemini_briefing_sync` functions, with the model routing in one place), and `briefing_context.py` (the eight-to-ten intelligence-source assembly). `routers/home.py` shrinks to roughly 1,200–1,500 lines and becomes actually a router again. The Lane 2A worker in `tasks/home_briefing_tasks.py` already lives outside the router and would just import from the new package.

The risk profile is medium. The briefing flow has many entry points (Lane 2A worker, admin refresh endpoint, request-path fallback, two Redis cache keys with subtle invalidation rules). The move has to preserve cache semantics exactly (`skip_cache=True` bypass of the legacy key, modern key write pattern, TTL) or stale briefings will appear in production. The existing test suite is strong for the briefing's output behavior but thinner on the cache path. The move itself is mechanical — function extraction into modules, imports rewritten — but a few sessions of careful validation after the move are essential.

**2. Consolidate the three plan generators into one supported path.** The problem is that three plan generation systems coexist in production code, each roughly 2,000+ lines, each with its own save path in `routers/plan_generation.py`, each with its own pace-order-violation validator, each used by live endpoints. `services/model_driven_plan_generator.py` (2,082 lines) is called by `/v1/plans/model-driven`, its preview endpoint, `services/ab_test_plans.py`, `services/coach_tools/plan.py`, eight tests, and six scripts. `services/plan_framework/n1_engine.py` (1,244 lines) is the V1 "N1 Engine V3" that the rebuild plan declared as the winner in March. `services/plan_engine_v2/engine.py` is the V2 that the founder coaching-reviewed on April 11 and that runs behind the `engine=v2` flag. The rebuild plan says "all legacy plan generators deleted" on March 29 with a 16,920 line deletion; the model_driven generator is still present and live.

The cost of the three-way split is that every plan concern (pace logic, volume contracts, taper construction, long-run progression, fingerprint consumption, KB rule evaluation) has up to three implementations, up to three test matrices, and up to three bug surfaces. The fingerprint bridge already feeds V1 and V2 but not `model_driven`. The `OUTPUT_METRIC_REGISTRY` lives in a separate module but is consumed by both plan paths differently. The wiki says V2 will become default after founder live testing and seven-day monitoring. When that happens, V1 will have a deprecation story but `model_driven` still won't.

The scope has two phases. Phase one is to trace every call site of `model_driven_plan_generator.py`, migrate each one to either V1 or V2 (depending on what actually serves each call site), and delete the module. That is a one- or two-session job because the call sites are enumerable (the grep I did above) and the replacement plan generator is already wired. Phase two is to decide on V1-vs-V2 cutover timing and then execute the V1 archive once the founder signs off. The archive itself is a quiet move — the `plan_framework/` directory stays, but `n1_engine.py` becomes a deprecation shim that raises if called, and the constraint-aware endpoint's default flips to V2.

The risk profile is higher than move 1 because V2 is not yet default and any cutover is a product event, not a refactor event. That is why this is structured as two phases: phase one deletes the truly legacy path with no product impact; phase two waits for the founder's V2 sign-off and then makes V1 → V2 the default. Doing phase one first also cleans the space so phase two only has to deal with two engines, not three.

**3. Unify the narration layer.** The problem is that nine different modules produce athlete-facing language: `intelligence/adaptation_narrator.py` (adaptation narration, 3A), `intelligence/moment_narrator.py`, `intelligence/narration_scorer.py`, `intelligence/narrative_memory.py`, `intelligence/narrative_translator.py`, `intelligence/neutral_terminology.py`, `intelligence/workout_narrative_generator.py` (3B, currently gated on narration quality), top-level `services/training_story_engine.py` (902 lines, race story / progression / connection detection for the Operating Manual), and inline LLM calls in `routers/home.py` (the briefing). Plus the coach's own LLM path in `coaching/_llm.py`. Plus the Operating Manual narrative generation in `routers/progress.py` (`_generate_narrative_llm`, `_generate_knowledge_llm`). No single module owns "what goes out to the athlete in words," so the Athlete Trust Safety Contract is enforced in one place, the KB violation scanner runs in another, the narration scorer runs in a third, the neutral terminology substitution runs in a fourth, and the briefing validators run in a fifth.

This is the hardest of the three moves because it touches every LLM-facing surface. The move is not to consolidate all the modules into one — each has a legitimate concern — but to create a single `services/narration/` orchestrator that every athlete-facing text surface goes through. The orchestrator would call the Translator, the Memory, the Terminology substituter, the Scorer, and the Violation checker in a fixed pipeline, and every surface (briefing, coach, Operating Manual narratives, adaptation explanations, home page, email digest) would produce its draft and then funnel through the orchestrator for the final pass. The modules themselves stay; what changes is that no surface is allowed to call the LLM, write to a JSONB field, or render to an athlete without running the pipeline first.

The scope is an interface extraction (one module: `narration/pipeline.py` with a single entry point) plus a migration of each surface to call it. The migration can be done per-surface over several sessions and does not require any surface to be rewritten. The risk is medium: the pipeline has to be permissive enough not to suppress content that is already passing through five surfaces today, which means the initial version has to be a near-passthrough that records what would have been changed. After a few weeks of observation, the pipeline gets teeth. That phasing is the same discipline the AutoDiscovery pipeline used — accumulate evidence, promote to active, then act.

All three moves continue the refactor's discipline: no new features, no new capabilities, no rewrites. They are moves, extractions, and one unification. Each one makes future feature work faster and safer. The order is by blast radius: move 1 is internal to the backend and has the clearest testable surface; move 2 involves product risk at phase two but phase one is clean; move 3 touches the largest number of surfaces and must be done in shadow mode first.

## Section 7 — What the refactor did not address

The refactor did three things cleanly: it retired `models.py`, it retired the `ai_coach.py` / `coach_tools.py` god files, and it pulled the sync and intelligence and coaching domains into subpackages with consistent backward-compat shims. None of that was easy and the execution is good.

What remains is the set of structural weaknesses the refactor did not touch.

The routers still hold business logic. Section 2 is specific about where: the home briefing, the plan save paths, the Operating Manual assembly. The refactor's axis was service organization; the router-to-service extraction is a different axis and was not attempted.

The top-level `services/` directory is still the kitchen sink. Roughly 120 real files (excluding the 33 shims) live there, many over 1,000 lines, many with overlapping concerns. Examples of the overlap: `shape_extractor.py` and `workout_classifier.py` and the in-router `_summarize_workout_structure` are three different classifiers of workout shape/structure running on the same activities; `training_story_engine.py` and `operating_manual.py` and `run_intelligence.py` and `insight_feed.py` all produce Operating Manual content from findings; `efficiency_trending.py` and `efficiency_analytics.py` and `efficiency_calculation.py` are three efficiency-related modules that a reader cannot rank by authority without reading all three. The refactor did not rationalize this; there was no easy natural split to apply the way there was for sync and intelligence.

The narration surface is fragmented across nine modules, covered in Section 6 already but worth naming here as a refactor gap: the refactor pulled narrative modules into `intelligence/` but did not unify them under a single pipeline, and that unification is a distinct piece of work.

The `coach_modules/` / `coaching/` / `coach_tools/` layering is not explained by naming. `coach_modules/` is an earlier extraction that the refactor left alongside the new `coaching/` package without folding it in. Either the three packages should be renamed to reveal the layering or `coach_modules/` should be absorbed into `coaching/` as internal modules. The current state is a working system but a confusing one to navigate.

The `model_driven_plan_generator.py` legacy is documented above. The rebuild plan declared it deleted; it is not.

The `provider_import/` package has one file. Either the package should be promoted (if more providers are coming) or the file should move to `sync/garmin_di_connect.py` and the empty package removed.

The `database.py` repo-root deprecation shim is still present. That is a minor cleanup item; it exists because some older call sites still import from `database` rather than `core.database`. The shim is harmless but the migration is incomplete.

The wiki itself has drifted. It still describes `ai_coach.py` as a 5.7K-line god file with "split deferred" and `coach_tools.py` as a 4.9K-line god file with "split deferred." Both were actually split. The `models.py` section in the wiki says "~85 models in models.py" even though `models.py` is gone and the models live in the eight-file package. The wiki's maintenance contract ("Every future code change ... should include: Wiki update") is the right discipline; this review is evidence that the discipline lapsed during the refactor itself, which is the worst possible time to miss a wiki update.

The `CONFOUNDER_MAP` and `DIRECTION_EXPECTATIONS` dictionaries at the top of `services/intelligence/correlation_engine.py` are hard-coded module-level dicts. Every new correlation input potentially adds entries. The current design (code change plus test for every confounder) is right in spirit; the mechanism (two big dicts at the top of a 2,822-line file) makes the contract less visible than it could be. A dedicated `services/intelligence/confounders.py` with the dicts, the selection function, and the tests colocated would make the contract structural rather than just textual.

## Section 8 — Observations not in scope

A handful of observations do not fit cleanly into the sections above but seem worth capturing so they are not lost.

The CI is set up to run the full backend suite nightly and fast smoke on push, which is sensible for a small team. The migration head check (`ci_alembic_heads_check.py`) expects `plan_engine_v2_001` as the single head; if the V1 → V2 consolidation proceeds, that constant becomes load-bearing for deploy correctness.

The `try/except ImportError` pattern around router registration in `main.py` (Garmin, GDPR, public tools, Runtoon, RPI, knowledge) is a quiet risk. If any of those modules accrues an import error that is not caught in tests, the route simply disappears from the API with only a warning log. Preferring explicit conditional imports (feature flag based) or hard failures would make the behavior less surprising.

The `correlation.py` model file holds both `AthleteFinding` (legacy) and `CorrelationFinding` (canonical). Their relationship is not commented in the model file. If `AthleteFinding` is kept for backward compatibility with older data only, saying so in a class docstring would make it clear which is authoritative for new code.

The `system.py` model file has become a grab bag. Billing, invites, Stripe, telemetry, Runtoon images, feature flags, ingestion state, experience audit, and promo codes all live there. As the product adds cross-domain concerns (telemetry, Runtoon, experience), this file will keep growing. If and when it crosses 1,500 lines it should be split — the natural seams are billing (Stripe, Subscription, Purchase), access (InviteAllowlist, FeatureFlag, ConsentAuditLog), ingestion (AthleteIngestionState, AthleteDataImportJob), and content (RuntoonImage, PageView, ToolTelemetryEvent).

The `OUTPUT_METRIC_REGISTRY` is the most important contract in the system that a new engineer would not know exists. Its presence in a class-level constant inside `n1_insight_generator.py` is functional but under-surfaced given the number of downstream consumers that are supposed to consult it. A dedicated `services/intelligence/output_metric_registry.py` module that holds the registry, the polarity lookup functions, and the 65 contract tests in a single directory would make this contract structural. This is a smaller version of the confounder-map observation in Section 7.

The `tasks/beat_startup_dispatch.py` pattern is the single most important piece of operational tribal knowledge in the system. It is documented in the wiki's infrastructure page, but a new engineer reading `celerybeat_schedule.py` in isolation would not learn from the schedule why daily tasks need a startup dispatch. A prominent comment at the top of `celerybeat_schedule.py` pointing at the startup dispatch file would save someone a multi-hour debugging session someday.
