## Part 1: Source-Derived Rules

### From Source A (Marathon Preparation)

**Rule A1: Four-Block Periodization**
```
Block 1: ENDURANCE (Base)
  - Build aerobic foundation
  - Introduce strides and hill sprints
  - NO threshold yet
  - Duration: 3-4 weeks

Block 2: LT + ENDURANCE (Build)
  - Threshold development
  - Maintain volume
  - Duration: 4-6 weeks

Block 3: RACE PREPARATION (Peak)
  - VO2max sharpening (sparingly)
  - Marathon-pace work
  - Duration: 3-4 weeks

Block 4: TAPER + RACE
  - Reduce fatigue
  - Maintain fitness
  - Duration: 3 weeks
```

**Rule A2: Marathon Success Factors (Priority Order)**
1. High Lactate Threshold → T-work is the core quality
2. Glycogen Storage + Fat Utilization → Long runs
3. Running Economy → Strides, hills, drills
4. High VO2max → Intervals (Block 3 only)
5. Quick Recovery → Easy running, sleep

**Rule A3: Weekly Structure**
- Rest or cross-train on recovery days
- Hard day → Recovery day (never back-to-back hard)
- Long runs on weekends
- Never 3 demanding days in 4-day span

**Rule A4: Masters Adaptations**
- Hill sprints and strides CRITICAL (maintaining stride length)
- Strength training 2x/week
- Easy runs should be SLOWER than younger runners
- May need extended schedule for recovery
- Sleep: 8-9 hours (vs 6-7 for younger)

**Rule A5: Taper**
- Duration: ~3 weeks for marathon
- Purpose: Correct accumulated wear, improve economy, repair damage
- Replenish glycogen, bolster immune system

### From Source B (Training Intensities)

**Rule B1: Volume Limits (% of Weekly Mileage)**

| Session Type | Maximum | Notes |
|--------------|---------|-------|
| Long Run | 25-30% | Max 150 minutes |
| Marathon Pace | 20% or 18mi | Whichever is less |
| Threshold | 10% | In one session |
| Intervals | 8% or 10K | Whichever is less |
| Repetitions | 5% or 5mi | Whichever is less |

**EXAMPLE for 50 MPW:**
- Long run max: 12.5-15 miles (but 150 min cap)
- MP max: 10 miles or 18mi → 10 miles
- T max: 5 miles in one session
- I max: 4 miles
- R max: 2.5 miles

**Rule B2: Weekly Distribution**
- Easy running: 65-80% of volume
- Quality sessions (T/I/R): 15-25%
- Marathon pace: 10-20% when marathon training

**Rule B3: Time-Based Limits**
- Long run: 150 minutes maximum
- Elite marathoners don't do 5+ hour training runs
- Use TIME, not just distance

**Rule B4: Recovery Philosophy**
- Stay at same weekly mileage 3-4 weeks before increasing
- Prefer step increases over gradual 10%/week
- Days off ARE training, not missed training

**Rule B5: Purpose First**
> "What is the purpose of this workout?" — If you can't answer, don't do it.

### From Source C (80/20 Intensity Distribution)

**Rule C1: The 80/20 Principle**
- 80% of training at low intensity (conversational)
- 20% at moderate-to-high intensity
- Elite athletes across ALL endurance sports converge on this
- This is optimal adaptation pattern, not coincidence

**Rule C2: The Trap**
- Recreational runners tend toward 50/50 (too much moderate)
- Creates chronic fatigue without recovery to adapt
- The "moderate hole" is the enemy

### From Source D (Cumulative Fatigue)

**Rule D1: Cumulative Fatigue Approach**
- Race day: 26.2 on tired legs
- 20 miles on fresh legs ≠ race simulation
- 16 miles after days of consistent running = better simulation
- Strategic weekly volume > single heroic long runs

**Rule D2: Long Run Cap (Cumulative Philosophy)**
- Some coaches cap at 16 miles (let fatigue do the work)
- Context determines: Fresh-legs runners need longer; cumulative-fatigue runners benefit from shorter

### From Michael's Data (Real-World Corrections)

**Rule M1: Age Is a Variable**
- PRs at 57 prove age predictions often wrong
- Individual data overrides age assumptions
- What matters: cumulative fatigue, recovery patterns, injury history

**Rule M2: Interval Timing**
- Intervals are NOT inherently dangerous
- Risk = intervals + high cumulative fatigue
- Safe: Intervals in BASE (fresh legs)
- Risky: Intervals in BUILD (high volume, fatigued)

**Rule M3: Speed Work Placement**
- Counterintuitive: Speed work in BASE phase
- Fresh legs + speed = adaptation without injury
- BUILD phase = T-work and MP (lower injury risk)

**Rule M4: Mid-Mileage Runners Need Frequency**
- 40-55 mpw runners: prioritize 6 days/week
- Goal: mechanical and aerobic durability
- Build base before aggressive workouts

**Rule M5: Easy Pace Calculators Are Too Fast**
- Standard tables too aggressive for easy running
- Use slower end of ranges
- Conversation test > calculated pace

## Part 3: Weekly Structure Rules

### Option A: Intensity Week + Easy Long (From TRAINING_PHILOSOPHY.md)

Use when: T-work is the week's quality session

```
Sunday:    Long run (easy)
Monday:    Rest / Gym / Cross-train
Tuesday:   Medium-long easy OR easy
Wednesday: Easy
Thursday:  QUALITY (T-work or hills)
Friday:    Easy
Saturday:  Easy (pre-long run)
```

### Option B: Easy Week + Workout Long

Use when: MP work is in the long run (no T that week)

```
Sunday:    Long run WITH workout (MP blocks)
Monday:    Rest / Gym
Tuesday:   Easy (recovery from LR)
Wednesday: Easy
Thursday:  Easy + strides (no T)
Friday:    Easy
Saturday:  Easy
```

### The Alternation Rule

From TRAINING_PHILOSOPHY.md:
> If there are two intense sessions during the week, the long run is easy.
> If only one intense session, the long run can also have a workout.
> Never three quality days.

**Implementation:**
- Week with T on Thursday → Long run easy (Structure A)
- Week with MP in long run → Thursday easy (Structure B)
- Alternate structures in Build 2 and Peak phases

## Part 5: Applying to Plan Generation

### Step 1: Determine Volume Tier
From athlete data or questionnaire:
- Low: <30 mpw (5K/10K) or <40 mpw (HM/M)
- Mid: 30-50 mpw (5K/10K) or 40-65 mpw (HM/M)
- High: 50+ mpw (5K/10K) or 65+ mpw (HM/M)

### Step 2: Apply Volume Limits
Calculate session limits based on Rule B1:
```python
long_run_max = min(weekly_miles * 0.30, 150/pace_minutes_per_mile)
mp_max = min(weekly_miles * 0.20, 18)
t_max = weekly_miles * 0.10
```

### Step 3: Select Phase Structure
Use 4-block structure from Source A with cutbacks.

### Step 4: Apply T-Block Progression
Scale based on volume tier:
- Low: 4x4min → 2x12min
- Mid: 5x5min → 25min continuous
- High: 6x5min → 40min continuous

### Step 5: Apply MP Progression
Scale based on volume tier:
- Low: 2mi → 6mi → 8mi
- Mid: 4mi → 8mi → 14mi
- High: 6mi → 12mi → 18mi

### Step 6: Assign Weekly Structures
Alternate Structure A (T-week) and Structure B (MP-long week) in build/peak.

### Step 7: Validate Against Checklist
Run through Part 4 validation checklist.

### 7.2 Architectural Root Causes (The Real Problems)

The 10 symptoms above stem from 6 deeper architectural failures.

#### ARCH-1: Phase spec metadata is dead code

`phase_builder.py` defines `allowed_workouts` and `key_sessions` per phase. `generator.py` **never reads them**. The generator runs its own independent if/else chains that partially and imperfectly replicate the phase spec.

**Result:** The framework's phase rules are documentation, not enforcement. Any drift in the generator's if/else logic produces plans that violate phase constraints (e.g., threshold in base phase, intervals in taper).

**Required fix:** `_get_quality_workout()` must select FROM `phase.key_sessions` as its primary constraint. The if/else sequencing logic must operate within that allowed set, not independently of it.

#### ARCH-2: T-Block progression is approximated with two types, not implemented

The framework specifies a 6-step weekly T-block progression (6×5min → 5×6min → 4×8min → 3×10min → 2×15min → 30-40min continuous). The code implements:
```python
if week_in_phase <= 2:
    return "threshold_intervals"
return "threshold"
```
Two workout types. No step sequencing. A week-1 threshold session is structurally identical to a week-6 threshold session. **The progression that makes T-work accumulate adaptive stimulus is entirely missing.**

**Required fix:** The T-block scaler must receive `(week_in_phase, total_threshold_weeks)` and produce sessions that step through intervals→continuous progression. Scale the tier-appropriate starting format (Low: 4×4min; Mid: 5×5min; High: 6×5min) and progress through the 6 steps proportionally to total threshold weeks.

#### ARCH-3: MP progression is wrong and MP-in-medium-long is missing

Two failures:

1. The framework requires MP work inside the **medium-long run** on specific weeks (e.g., Week 9: "4mi in medium-long, 2×2mi MP blocks"). The code only assigns `long_mp` to the Sunday long run. The Tuesday medium-long never contains race-pace work under any normal condition. This eliminates roughly half the MP accumulation opportunities the framework specifies.

2. Total cumulative MP mileage (required: 40-50+ miles before race day) is not tracked or enforced. A plan can generate 15mi total MP work and pass all current tests.

**Required fix:** Build an `MPProgressionPlanner` that takes `(tier, total_mp_weeks)` and returns a week-by-week sequence specifying: which weeks get MP-long-run, which get MP-medium-long, target MP miles per session, and whether the week is Structure A or B. Track cumulative MP miles and validate against the 40-50mi minimum.

#### ARCH-4: Cutback weeks are positional math, not structural

The framework places explicit cutback weeks at phase transition boundaries (Week 4, 8, 12 in 18-week plan). The code uses `week % cutback_freq == 0` — this accidentally aligns for 18-week plans but drifts for shorter plans. More importantly, cutback weeks have no phase identity: they produce reduced volume but the same workout-type logic as surrounding weeks.

A proper cutback week: keep the key quality session at reduced volume, eliminate all secondary sessions, reduce easy volume. The current code sometimes eliminates quality entirely on cutback (returning `easy_strides` for marathon/half), which is worse than a properly reduced quality session.

**Required fix:** Cutback week logic must: (a) maintain the phase's primary key session at 70% of normal distance, (b) eliminate secondary quality (not primary), (c) fall at phase transition boundaries rather than every Nth week.

#### ARCH-5: WEEKLY_STRUCTURES is static — Structure A/B alternation is incomplete

The framework defines two weekly structures that must alternate in build/peak phases:
- **Structure A (T-week):** Thursday = threshold, Sunday = easy long
- **Structure B (MP-long week):** Thursday = easy + strides, Sunday = MP long run, **Tuesday = easy (not medium-long)**

The code correctly converts Thursday to `easy_strides` on MP-long weeks via the `is_mp_long_week` flag. But the Tuesday slot always remains `medium_long` regardless. On a Structure B week, Tuesday medium-long after a Sunday 20-mile MP long run violates the recovery principle: **the athlete needs easy running after the big workout, not a medium-long effort.**

Additionally, the 55mpw threshold for secondary quality in race-specific phases (`weekly_volume >= 55`) cuts off competitive athletes at 45-54mpw who are clearly in the race-specific phase with two quality sessions.

**Required fix:** Apply `is_mp_long_week` to the Tuesday slot too: convert `medium_long` → `easy` on Structure B weeks. Remove or lower the 55mpw gate for secondary quality in race-specific phase (gate should be experience-level + phase, not raw mileage).

#### ARCH-6: Constraint-aware generator is a completely disconnected system

`constraint_aware_planner.py` and `workout_prescription.py` are an entirely separate implementation that shares no code with `PhaseBuilder`, `WorkoutScaler`, `WEEKLY_STRUCTURES`, or any periodization logic from this framework. It implements its own theme rotation without a volume build curve, producing plans that oscillate at fixed volume every 3 weeks with no progressive overload.

**This is the most severe failure for competitive athletes** — the constraint-aware path is the highest-tier output but produces the worst plans.

**Required fix:** Either (a) refactor the constraint-aware path to use `PhaseBuilder` + `PlanGenerator` with `FitnessBank` as the N=1 overlay — the correct long-term solution — or (b) add proper volume build curve, phase-aware workout selection, and T-block/MP progression to the existing constraint-aware code as a bridge. Option (a) is the correct architecture. Option (b) is a minimum-viable patch.

### 7.4 N=1 Overrides (Non-Negotiable)

These athlete-history signals must override framework defaults, not be ignored by them:

1. **`l30_max_easy_long_run_miles`**: If athlete's recent long runs exceed `LONG_RUN_PEAKS`, the peak cap must be elevated. A runner who regularly runs 18mi long runs does not get a 14mi cap because a table says so.

2. **`current_weekly_miles`**: Volume tier and all session proportions are derived from this. It must be the baseline, not `tier_params.min_weekly_miles`.

3. **`days_per_week`**: The athlete told the system how many days they run. This is respected absolutely. No plan generates more running days than requested.

4. **`experience_level`**: MP long runs, secondary quality sessions, dress rehearsals — all gated on experience. A beginner does not receive dress rehearsal workouts in week 2.

5. **`race_history_rpi`**: When present, this calibrates every pace zone. A plan built without this is an estimate; a plan built with it is personalized.

### 7.5 What Must Be Tested After Every Plan Generation Change

Any change to plan generation code must pass these assertions across the full athlete matrix (10 archetypes × 4 distances × standard/semi-custom/constraint-aware):

1. `medium_long_miles < long_run_miles` — every week, no exceptions
2. `min(workout.miles for all workouts) >= 0.0` — no negative mileage
3. `workout_count_per_week == days_per_week` — ±1 for race week only
